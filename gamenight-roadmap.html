▾How this roadmap works

It tells you what to build, in what order, why, and when to commit and push — but not how. Every phase starts with “Learn first” links so you discover the actual code yourself. The parts marked Design decision are the ones worth struggling with — that's where the real learning is. No code is included on purpose; ask for code only when you're genuinely stuck.

  • Work top to bottom. Each phase is a shippable slice the next one builds on — don't jump ahead.
  • Tick the squares as you go — this is your progress tracker.
  • Read the Learn first links before writing anything in that phase.
  • The commit chips tell you exactly when to commit and give you a ready-made message.
  • Design decision notes are mini-assignments: decide, write down your reasoning, then build.
  • Pro practice lines are the real-world / DevOps habit for that phase — what a senior engineer would insist on.
▾Git workflow — read once, apply every phase
  • Branch per phase. Create a branch like feat/phase-2-auth off main, do the phase's work there, then open a Pull Request into main and merge it — even solo. Reviewing your own PR is a great habit. GitHub Flow
  • Commit small and often. One atomic commit per logical change (≈ one square). A commit should do one thing and still leave the app working.
  • Use Conventional Commits: type(scope): short description in the imperative. Types: feat, fix, refactor, chore, docs, test, style, perf. Spec · Why it matters
  • Push at the end of every work session, and always at the end of a phase.
  • Tag releases at the milestones using semantic versioning (v0.1.0, v0.2.0…). Pro Git: Tagging (one short chapter)
  • Never commit secrets. Write your .gitignore before your first commit (ignore node_modules, .env*, build output, .expo). github/gitignore
If you forget everything else: commit per logical change, push at the end of each session, tag at each milestone.

▾Tech stack & why

You picked “some React Native experience,” TypeScript, and lowest-cost backend. Given that:

  • Expo (latest SDK) + Expo Router + TypeScript — file-based routing that works on iOS, Android, and web from one codebase. Expo Router · Expo + TypeScript · Expo changelog
  • Supabase — Postgres + auth + realtime + storage + functions on one free tier. Postgres fits because almost everything you track is relationships and aggregates. Supabase docs

▾DevOps & Docker — applied honestly

Real-project habits, used where they actually help — not cargo-culted:

  • Docker does NOT wrap the app. React Native / Expo builds use native toolchains (locally or on EAS) — don't try to containerize the mobile app itself.
  • Docker DOES run your backend. The Supabase CLI launches Postgres + Auth + Storage + Realtime in containers — real prod-parity local dev, plus local Edge Function testing and an identical database in CI.
  • Everything is code: schema as migrations, pipelines as YAML, config as .env.example — all reviewed in PRs.
  • Quality enforced by tooling: pre-commit hooks block bad commits, CI blocks bad merges, branch protection makes it non-optional.
  • The mantra: local = CI = prod. Same containers and migrations everywhere, so “works on my machine” disappears.
Set up in Phase 0.7 (foundations); fully automated in Phase 11 (CI/CD).

▾Data model overview — think it through before Phase 1

You don't need SQL yet — just understand the shapes:

  • Profile — one per user (display name, avatar), linked to the auth user.
  • Friendship — a link between two profiles with a status (pending / accepted).
  • Party — created by a host; just a name and a fixed roster. The when / where / what does NOT live here.
  • Party roster — the players in a party. This is your hardest design problem.
  • Game night — one evening inside a party; carries its own date/time, location, and games played. Scheduling lives here, not on the party.
  • Game — a catalog entry (BGG or custom) with type, player count, difficulty, and a scoring schema.
  • Game result — links a game night + a game + structured per-player data (points, bonuses, placement, money…) plus the computed winner.
  • Scoring schema — per game: a win-condition archetype + the fields to capture. Sources: hand-curated, mechanic-based default, AI-generated, or override.
  • Message — chat between friends (and maybe party group chats).
Decision (locked in) — the immutable roster: a party's player count and identities are fixed once set so scores stay comparable. You've chosen to enforce this in the database — the strongest option, because the rule holds no matter what touches the data. The UI can hide the buttons, but the database is what guarantees it.

Implementation shape (work out the SQL yourself): give a party a locked state, then add a Postgres trigger (backed by matching RLS) that rejects any insert/delete on roster rows, and any change to a player's identity, once locked. When to lock? Recommended: lock the moment the first game night is created — fix typos before scores exist, frozen forever after real play begins.

Learn the tools: Postgres triggers · Row Level Security · Postgres constraints. Sketch the model first with dbdiagram.io.
