The app, in plain language — read this first, Denis

Gamenight is a free phone + web app for friend groups that have regular board-game nights.

You create a party — your fixed group of players. Then every time you get together you log a game night: the date, where you are, which games you played, and everyone's scores. The app works out who won and keeps score across the whole group over time — overall win rates, per-game win rates, and head-to-head ("am I actually better than you at Catan?"). It also has a searchable catalog of board games (real data from BoardGameGeek), a friends list, real-time chat, and push notifications for invites.

Think of it as a stats tracker + social hub for game nights — it remembers every result so the group can finally settle who's the best.

  • Create a party (a fixed roster) so scores stay comparable.
  • Log a game night: date, place, games played, each player's score.
  • See who won and win rates over time (overall, per game, head-to-head).
  • Search a catalog of board games, or add your own.
  • Add friends, chat in real time, get notified about invites.
Built with (all new to you is fine): Expo / React Native + TypeScript (the app, phone + web from one codebase), Supabase (the backend — database, login, realtime), GitHub (code + tickets + automation), and Docker (runs the backend on our laptops). Everything has a free tier — total cost to build & run: $0.

How we work — the important part

The goal is that both of us can do everything by the end. That doesn't happen by accident — we build it in with six habits:

  1. Pair programming. For anything new or hard, we work on one screen: one drives (types), the other navigates (thinks ahead, looks things up). Swap every ~30 min. Fastest way to move knowledge between two heads.
  2. Whoever didn't build it, reviews it. Every change goes through a Pull Request the other person reads and approves. You can't approve what you don't understand — so reviewing is forced learning.
  3. Teach-back. After each phase, whoever led it gives the other a 10–15 min walkthrough.
  4. Rotate the scary stuff. Teo leads the first hard thing (DB security, Phase 3); Denis leads a similar thing later (Phase 7) with support. Nobody "owns" a mystery area.
  5. Everything is a ticket. No work happens unless it's on the board. Denis owns the board.
  6. Shared learning log. A running doc where we both jot what we learned and the links that helped.

Denis's ramp-up: zero to co-developer

You do not need to know how to code to be useful from day one. You start by owning the parts of a real team that aren't deep coding, learn fundamentals on the side, and pair every day until you're writing features yourself.

Weeks ~1–3 · Own the process, learn the basics

Stage 1

  • ~1.5 h/day learning: a beginner JavaScript / TypeScript + React course (freeCodeCamp, free) and Git basics.
  • Own the GitHub Project board: create it, write the first user stories, keep it tidy.
  • Help with design: wireframes in Figma (free) + choosing colors/spacing. Phase 0.5 is visual — a great first win.
  • Pair ~1–1.5 h/day with Teo: watch, ask "why," start driving simple bits.
Weeks ~4–10 · Training wheels

Stage 2

  • Take "good first issues" — small, clearly-described coding tasks Teo prepares (a form field, a list screen, a style fix, a small test).
  • Own QA: test the app, write down what you tried, file bugs as tickets.
  • Own catalog content (Phase 6): add games + define how each is scored — valuable, not-deep-code work.
  • Keep running the board and sprint planning.
Weeks ~10+ · Co-developer

Stage 3

  • Lead a whole feature solo. Phase 7 (Friends) is your "graduation" — it reuses patterns you've already seen.
  • Review Teo's Pull Requests. From here we rotate freely and genuinely both know the app.

Tickets — how we track work (Agile-lite, free)

Tool: GitHub Issues + GitHub Projects — built into the repo, free, and it talks to our pipeline. Denis owns it.

Every ticket has a type, an assignee, a size (S / M / L), and acceptance criteria (a short "done" checklist).

Definition of Done: code written → tests pass in CI → the other person reviewed & approved the PR → merged → (if a Story) demoed.

Sprints are one week. Monday = planning (pull tickets into the sprint). Friday = quick demo + 10-min retro. Denis runs both — the best PM training there is. Write Closes #42 in a PR and GitHub moves the ticket to Done automatically on merge.

Our daily & weekly rhythm

Every day: a 10-min standup (Denis runs it — what I did / doing / blockers), ~1–1.5 h pairing overlap, then solo on your own tickets.

  • Denis (~4 h): ~1.5 h learning · ~1.5 h board + design + QA + pairing · ~1 h a good-first-issue.
  • Teo (~8 h): ~5–6 h building · ~1 h reviewing Denis's PRs + prepping his next tickets · ~1 h pairing/teaching.

Every week: Monday planning · Friday demo + retro. One sprint ≈ one phase (give or take).

Bottom line: something usable & demo-able in your hands in ~6–8 weeks, and a polished, automated v1.0 release in roughly 4–5 months.
Want a real release sooner? Scope a leaner "v1": ship auth + parties + game nights + basic stats (Phases 0–5) and push chat, push notifications, and the full catalog to "after v1." A genuinely shippable app in about 2–2.5 months.

Faster: small tickets, pair on the scary stuff, don't gold-plate v1.   Slower: skipping tests early, big unscoped tickets, long gaps between sessions.

Is "both of us know how to do everything" possible?

Yes — and it's the most valuable thing you'll get out of this. It just has to be deliberate. The pairing, the "whoever didn't build it reviews it" rule, the teach-backs, and rotating the scary stuff are specifically there so neither of us ends up the only person who understands a part of the app. By v1.0, Denis won't be "the non-programmer" anymore — he'll have written, reviewed, and explained code across every layer. That's the plan.
